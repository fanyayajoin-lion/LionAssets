## 核心定位

| OpenClaw（前台閘道） | Hermes（後台大腦） |
|---|---|
| 多渠道統一入口 | 長周期複雜任務 |
| 輕任務即時響應 | 長期記憶 & 技能沉淀 |
| 會話路由 & 權限隔離 | 定時無人值守調度 |
| 結果美化輸出 | 自迭代優化 |

## 黃金協同邏輯

```
用戶多渠道發指令 → OpenClaw 識別意圖
     ↓
簡單輕任務 → OpenClaw 直接處理
複雜長流程 → 封裝標準化任務包，透過 ACP 協議轉發 Hermes
     ↓
Hermes 完成運算 → 結構化原始資料回傳 OpenClaw
OpenClaw 排版、脫敏、適配渠道格式 → 推送給發起用戶/業務群
```

## 三種部署模式

### 模式 1：輕量單機混合（5 人以內）
- 同一台伺服器跑 OpenClaw + Hermes
- 適用：自媒体、小型銷售團隊、獨立 SaaS 開發者
- 配置：Hermes 限制最大 CPU 佔用 50%

### 模式 2：分離雙機生產架構（10-50 人）
- 閘道伺服器（2 核 4G）：僅運行 OpenClaw
- 執行伺服器（4 核 8G+）：獨立部署 Hermes
- 適用：電商客服、銷售線索挖掘、企業自動化運維

### 模式 3：企業級擴容（50 人以上）
- OpenClaw 集羣負載均衡 + Hermes 多 Worker 分散式任務池
- 支援多部門獨立業務流程隔離、權限分級

## 六大落地商業場景

### 場景 1：銷售線索自動化獲客 & CRM 管理
- 流程：抓取 → 清洗 → 分層 → 導入 CRM → 生成跟進話術
- 收益：銷售每日省 3-5 小時手動找線索

### 場景 2：客戶售後客服全流程自動化
- 流程：OpenClaw 分流 → 簡單問題本地回覆 / 複雜問題轉 Hermes → 每日早 8 點推送復盤報表
- 收益：客服即時回覆 + 自動化復盤

### 場景 3：企業內部運營 & 數據日報自動化
- 流程：一句話指令 → 多平台 API 取數 → Hermess 合併計算 → 自動發送管理層群
- 收益：無需人工彙報

### 場景 4：內容營銷批量生產 & 多平台發布
- 流程：需求下發 → Hermess 批量創作 → OpenClaw 定時自動分發
- 收益：一鍵產出 + 自動發布

### 場景 5：研發/IT 運維自動化
- 流程：指令 → OpenClaw 輕量查詢 → Hermess 重型分析 → 自動告警
- 收益：24 小時定時巡檢、自動修復

### 場景 6：競品 & 行業情報監控
- 流程：Hermes 定時抓取 → 整合對比分析 → OpenClaw 推送精簡情報摘要
- 收益：每日早間自動推送

## 任務分流標準（關鍵）

### OpenClaw 獨立處理（不轉 Hermes）
- 單次短問答、產品基礎問答
- 單文件讀取、單次網頁查詢、單條郵件收發
- 即時客戶對話、簡單表格導出、即時翻譯

### 必須轉 Hermes 執行
- 批量處理（≥50 條數據、多文件合併）
- 多步驟串聯流程、跨 3 個以上系統取數
- 定時循環任務、長文本深度分析、代碼生成調試
- 需長期上下文記憶、復用歷史業務模板的工作

## 商用避坑要點

1. **不要把全部任務丟給 Hermes**：大量簡單諮詢會造成閘道排隊、客戶回覆延遲
2. **分離部署生產環境**：單機僅適合測試/小微業務，商用雙機隔離保障服務穩定
3. **Hermes 批量任務設置執行上限**：超時自動終止，防止無限循環消耗伺服器資源
4. **OpenClaw 統一美化輸出**：禁止直接推送 Hermes 原始日誌給客戶/管理層

## 業萱的洞察

- 我的架構已經接近這個模式的 80%
- 缺的是「任務分流標準」和「結果美化輸出」
- 我的優勢：我就是閘道，不需要額外的 OpenClaw 層
- 我要學的是：讓 AI 知道什麼情況要轉、什麼情況自己處理
