# 群獅會議紀錄 — 碎片 SLA 與健康檢查優化

## 背景

群獅週會討論碎片自動入庫停擺 3 天的根因與優化方案。雙方達成共識：主因為「條件式每日知識整理」cron 輸出超長截斷，非執行遺漏。

## 根因分析

- 7/16 19:57 之後 raw/ 無新進度，與 cron truncation 時間點吻合
- cron prompt 過長，AI 輸出超過限制導致 RuntimeError
- 沒有主動警報機制，積壓直到寫日記才被发现

## 決議事項

### 1. cron prompt 重構
- 增加 3000 字輸出上限
- 精簡報告格式，只列已處理清單
- 無新資料時只回傳「無新來源，不需整理」

### 2. 碎片 SLA 三層機制

| 層級 | 觸發條件 | 動作 |
|------|---------|------|
| 🟡 黃燈 | raw/ ≥ 3 筆 或 最老碎片 > 12h | 健康檢查標黃 |
| 🔴 紅燈 | raw/ ≥ 5 筆 或 最老碎片 > 24h | 每日回報標紅 + 通知秘書獅 |
| 🚨 警報 | raw/ ≥ 8 筆 或 最老碎片 > 48h | 主動 Telegram 報警給業萱 |

### 3. 健康檢查三欄位監控
- raw/ 待處理數量
- 最老碎片年齡（小時）
- 最近一次知識整理 cron 狀態

### 4. 統一計數標準
- `ls inbox/raw/ | wc -l`（排除 .git、hidden files）
- 積壓時效以最老檔案 mtime 為準

## 落實狀態（2026-07-18 管理獅執行）

- ✅ cron prompt 已重構，增加輸出限制與精簡格式
- ✅ 健康檢查腳本已加入三欄位監控 + SLA 嚴重度判斷
- ✅ 積壓碎片已清理（3 筆移至 processed）
- ⏳ 萃取日誌吞吐量追蹤欄位 — 待下次每日整理實裝
