# 📘 碎片的接收、整理、入庫 SOP

## 問題情境
業萱透過 Telegram 丟進碎片（截圖、網址、文字、對話紀錄），需要自動判斷、整理、入庫。

## 執行步驟

### Step 1: 接收碎片
- 記錄來源類型：截圖 / 網址 / 文字 / 對話
- 記錄日期時間
- 存入 `inbox/raw/`（不修改、不刪除）
- 命名規則：`inbox/raw/YYYYMMDD-HHMMSS-<短識別>.md`

### Step 2: 自動判斷（混合模式預判）
收到訊息後先判斷類型：

**直接過濾（不存）**：
- 問候語、確認存在、測試訊息
- 純問題可直接回答
- 無意義對話、情緒宣洩

**詢問是否存**（存入 inbox/raw 後問業萱）：
- 網址、文章、YouTube 影片
- 截圖、圖片、文件
- 工具、軟體、資源分享
- 品牌、公司、人物、產品
- 商業策略、方法論、框架
- 經驗分享、案例

### Step 3: 預摘要 + 標記來源
詢問業萱是否儲存後，若同意：
1. 建立 Markdown 中繼資料檔
2. 寫入 frontmatter：source_id, source_type, source_url, captured_at, title, excerpt
3. 摘錄重點（≤200 字）

### Step 4: 搜尋 wiki 連結
1. 在 wiki/ 中搜尋是否有相關頁面
2. 若有：更新現有頁面，追加引用
3. 若無：標記「新主題候選」，準備建立新頁面

### Step 5: 存入 inbox/processed/
1. 建立「老闆理解版」摘要
2. 放入 `inbox/processed/YYYY-MM-DD-對話摘要.md`
3. 回傳 Telegram 通知業萱

### Step 6: 回流 wiki（管理獅職責）
1. 依內容類型放入 wiki/concepts/ 或 wiki/entities/ 或 wiki/cases/
2. 建立雙向連結（outbound ≥2, inbound ≥1）
3. 更新 index.md
4. append log.md

## 注意事項
- `inbox/raw/` 永遠不可修改、不可刪除
- 碎片接收是自動化流程，不需要業萱參與判斷
- 只有「詢問是否存」的情況需要業萱確認
- API Key 只記錄需求與欄位，不得寫入實際 key
