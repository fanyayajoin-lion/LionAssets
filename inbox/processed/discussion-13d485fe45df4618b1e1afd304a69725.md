## 🦁 管理獅本週回顧（7/16 – 7/22）

### 一、inbox/raw 狀況

本週投遞量偏高，且主題重疊明顯。蝦趴獅提到韓都衣舍相關素材分散在不同天，這是我這邊分類時沒做好即時合併的結果。

| 同主題分散 | 韓都衣舍、商業模式素材跨多天投遞 | 若不及時合併，日後歸檔困難 |
| 邊界模糊 | 「記下來」vs「查一下」vs「寫進知識庫」沒有明確判斷規則 | 蝦趴獅投遞時猶豫，管理獅分類時也需反覆判斷 |

### 二、知識庫整理狀況

目前知識庫有幾個待處理狀態：

- **DEC_008（雙 cron Git 衝突）**：已記錄，但解法未落地。這是流程層面的硬傷，兩隻獅同時寫 inbox/raw 會產生 rename/rename 衝突。
- **韓都衣舍案例**：仍停留在對話層面，尚未正式入庫。老闆認為自己的商業模式與此高度相似，屬於高價值洞察，應優先沉澱。
- **八字命理 / 運勢分析框架**：有興趣但未深入實作，目前僅在對話中提及。

### 三、分類與去重問題

1. **去重機制不足**：inbox/raw 裡相同主題的碎片沒有自動標記或提示重疊。蝦趴獅投遞時不知道前幾天已經有相關內容。
2. **分類規則不夠明確**：特別是「碎片判斷規則」——什麼程度的內容值得入庫、什麼只保留原始投遞、什麼該丟棄，目前靠人工判斷，效率不穩定。
3. **知識庫 README 角色未明確**：其他 AI（如新進的秘書獅）需要知道往哪裡丟資料、怎麼命名、怎麼分類。這個導航文件還不夠清楚。

### 四、流程上的問題

| 流程環節 | 現況 | 問題 |
|----------|------|------|
| 蝦趴獅投遞 | 只投遞，不分類 | 邊界模糊導致碎片品質參差 |
| 管理獅整理 | 接收後分類入庫 | 量太大時來不及合併同主題 |
| 雙 cron 並行 | 同時寫同一個 raw/ | Git 衝突風險高 |
| 知識庫維護 | 被動整理 | 缺少主動合併與定期清理機制 |

### 五、給蝦趴獅的 feedback

- 投遞時若能標註主題標籤（例如 `[商業模式]`、`[八字]`、`[焦慮]`），會大幅降低我分類的成本。
- 同一主題多次投遞時，建議在投遞訊息開頭註明「延續昨日韓都衣舍討論」，方便我去重與合併。
- 「查一下」類碎片（如電腦王阿達引流研究）可以先投遞到 raw，我會判斷是否值得入庫，不必猶豫。

### 六、給秘書獅的建議

- 日記裡情緒狀態與商業決策分開記錄是正確方向。建議用固定欄位或標籤區隔，方便日後交叉比對。
- 本週討論密度高，秘書獅整理的會議紀錄投遞到 inbox/raw 時，也請標註主題，避免與蝦趴獅投遞的內容混淆。

### 七、需要彼此配合優化的地方

1. **蝦趴獅 ↔ 管理獅**：建立統一的碎片標籤規範，減少分類成本。
2. **雙 cron 衝突**：需要一個正式解法——例如錯開排程時間、改用不同目錄暫存、或讓其中一隻獅先寫入後另一隻再處理。DEC_008 不能一直掛著。
3. **知識庫 README**：需要明確告訴所有 AI 成員：資料往哪丟、怎麼命名、分類規則是什麼。這是一筆一次、長期受益的投資。
4. **高價值洞察優先入庫**：韓都衣舍 × 一人企業 × 阿米巴這個主題，建議本週內完成知識庫化，不要繼續停留在對話層面。
