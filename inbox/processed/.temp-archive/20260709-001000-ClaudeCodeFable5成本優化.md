# Claude Code 不一定要全程開 Fable 5 — 兩種省錢又高效的玩法

來源：Telegram 直接輸入（業萱分享）

## 方法一：Advisor 模式

先切換主模型：

```
/model sonnet 
```

再開啟 Advisor：

```
/advisor fable
（或 /advisor claude-fable-5）
```

運作方式：

- Sonnet 5 負責 90% 以上的實際執行
- Fable 5 只在關鍵決策、架構設計、卡關時提供建議

也就是：**Fable 思考，Sonnet 執行。**

Anthropic 在 SWE-bench Pro 的測試：

- Sonnet 5 + Fable 5 Advisor ≈ Fable 5 92% 的表現
- 成本只要 63%

## 方法二：Orchestrator 模式

讓 Fable 5 擔任 Orchestrator（編排者）：

- 規劃任務
- 拆解工作
- 指派給 Worker（Sonnet 5）
- 最後整合結果

大量 Token 由 Sonnet 5 消耗，成本大幅降低。

Anthropic 在 BrowseComp 的測試結果：

- Fable 5（Orchestrator）＋ Sonnet 5（Worker）
- 達到 Fable 5 96% 的效果
- 成本只需要 46%

簡單理解：**🧠 Fable 負責想，⚙️ Sonnet 負責做**

## 實踐方法

### 方法一：建立 /orchestrator 指令

在 `~/.claude/commands/orchestrator.md` 建立 Custom Command：

內容讓 Claude 進入 Orchestrator 模式：

- 負責整體規劃與拆解
- 執行工作委派給 Sub-agents / Workers
- 最後負責審核與整合
- 優先使用低成本模型完成細節
- 保持 Context 精簡

使用方式：`/orchestrator 幫我做登入功能`

### 方法二：寫進 CLAUDE.md 預設

在專案的 CLAUDE.md 加上：

```markdown
## 預設工作模式：Orchestrator

你預設使用 Orchestrator 模式：
- 你負責整體規劃、任務拆解、委派與最終審核
- 把執行細節委派給子代理（worker），優先使用 Sonnet 5 等低成本模型
- 保持上下文精簡，只保留重要決策與總結
- 每個子任務都要明確定義驗收標準
- 盡量平行委派加速
```

## 核心心法

> 🧠 Fable 負責想
> ⚙️ Sonnet 負責做

強模型做判斷、弱模型做執行 — 這跟知識庫已有的「AI 蒸餾與分層使用法」理念一致。
