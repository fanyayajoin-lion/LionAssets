# 群獅靈感知識庫 AI 工作規則

你要稱呼使用者為老闆。

這是群獅的靈感知識庫，從靈感出發，經過收集、整理、蒸餾、記憶、輸出、回流，慢慢變成可轉移的數位第二大腦。

## 老闆身份與語氣

老闆是業萱。

核心定錨句：

> 有實業背景的商業 AI 摸索者。不是來教你的，是分享我自己走過的路。

對外介紹可依場合使用：

- 對傳產老闆：我是業萱，做了近 8 年連鎖品牌營運，協助 50 幾家企業轉型，現在在研究怎麼用 AI 幫老闆把自己從公司裡解放出來。
- 對電商創業者：我是業萱，研究商業模式 3 年、自媒體 2 年、AI 1 年，幫一人公司用 AI 建系統，讓你不用自己做所有事。
- 對媒體或演講場合：我是樊業萱，遠特企業管理諮詢負責人、群獅整合行銷創辦人，專注幫有實業的創業者用 AI 建立可複製的商業系統。

核心信念：

- 工具會變，底層邏輯不會變。
- AI 是放大器，商業框架不強，AI 只會放大問題。
- 用 ESBI 視角幫有實業的老闆從 S 象限走向 B 象限。
- OPC 一人公司會變多，B2B2C / S2B2C 是大方向。
- 真誠分享，不包裝。

## 核心定位

這個知識庫不綁 Codex，也不綁 Claude Code。所有規則都要寫成通用 LLM 可讀。

每次任務的標準流程：

```text
讀 index.md -> 收集/讀取資料 -> 補全背景 -> 整理 wiki -> 蒸餾 synthesis -> 評 Lion Score -> 產出 output -> 回流 -> 寫 log.md
```

## 必守規則

- 查詢前必須先讀 `index.md`。
- 操作後必須 append `log.md`。
- `raw/` 永遠不可修改、不可刪除。
- `outputs/` 必須回流到 `wiki/` 或 `wiki/synthesis/`。
- workflows、skills、outputs、assets 不會自然更新，必須主動檢查並寫入。
- API Key 只記錄需求與欄位，不得寫入實際 key。
- 若需要最新資料，必須連網查證並保留來源；查不到可靠資料時標記「需查證」。

## 資產優先原則

同一主題重複出現三次以上，禁止永遠停留在 Note。必須判定是否升級為：

1. Skill
2. Asset
3. Memory

判斷方式：

- 可被重複調用的流程：升級為 `skills/`
- 可複製給不同客戶或內容使用的模板/素材：升級為 `assets/` 或 `wiki/synthesis/`
- 會影響老闆長期決策與偏好的資訊：升級為 Memory 類紀錄，並寫入 `wiki/synthesis/` 或相關條目

## Lion Score

每篇整理後內容都要加入：

```yaml
lion_score:
  變現價值: 1-5
  可複製性: 1-5
  長期價值: 1-5
  顧問可用性: 1-5
  內容延展性: 1-5
```

評分原則：

- 變現價值：能否用於顧問、課程、提案、產品或內容轉單。
- 可複製性：能否套用到不同品牌、傳產老闆、電商或一人公司。
- 長期價值：是否不受短期工具或新聞影響。
- 顧問可用性：能否變成診斷、框架、提問、SOP。
- 內容延展性：能否延伸成文章、短影音、簡報、直播、課程單元。

## 品牌案例規則

遇到品牌案例 PPT/PDF，不可只摘要原檔，必須先補資料再歸檔：

1. 預設不複製原始檔；保留原始路徑，將摘要、補全結果與歸檔卡寫進知識庫。
2. 只有老闆明確要求保存原檔時，才複製到 `assets/brand-cases/品牌名/`，且不得覆蓋。
3. 讀取原檔，整理原 PPT 重點與使用的商業框架。
4. 補充品牌最新狀態、營收、產品線、通路、私域、自媒體、直播、AI/數位化應用。
5. 若資料有年份差異，保留「原 PPT 觀點」與「最新補充」兩層。
6. 產出 `wiki/cases/產業分類/品牌名.md` 與 `wiki/entities/品牌名.md`。
7. 蒸餾可複製打法、可延伸商業模式、對傳產老闆與 OPC 的啟發。
8. 更新 `index.md`、`hot.md`、`log.md`。

## 任務結束自我檢查

- 是否要更新 `index.md`？
- 是否要更新 `hot.md`？
- 是否 append `log.md`？
- 是否產生可重複 workflow？
- 是否有 skill candidate？
- 是否有 output 要歸檔？
- 是否有 asset 要連回 output？
- 是否有孤島、矛盾、過期內容？
